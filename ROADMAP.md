# HandOfFateAccess - Build Roadmap

Phased plan. Each phase ends with a concrete, verifiable proof in the real game before moving on. Check items off as completed. See `CLAUDE.md` for architecture and conventions.

## Phase 1 - Plugin, logging, and speech (prove the whole output chain)
Stand up the full speech and logging stack in one go, so the first run proves output works end to end, not just that a DLL loaded. There's no value in a bare loader milestone on its own.
- [x] Install BepInEx 5.x **x86** into the game folder; launch once to generate `BepInEx/` config. (Installed BepInEx 5.4.23.2 x86; config generates on the first launch.)
- [x] Create the plugin project (`HandOfFateAccess.csproj`), target **.NET 3.5**, reference BepInEx, 0Harmony, and the game's `Assembly-CSharp.dll` / `UnityEngine.dll` / NGUI types. `[BepInPlugin]` `BaseUnityPlugin` entry.
- [x] `build.ps1`: build + copy the plugin DLL and the x86 Tolk DLLs (+ companions) into `<game>\BepInEx\plugins\`.
- [x] Logging setup: a mod `Log` helper writing to the BepInEx logger / Unity log, every line prefixed `[HoFAccess]`.
- [x] Speech: lift `Speech/` from oni-access (`ISpeechBackend`, `TolkBackend`, `SpeechEngine`, `SpeechPipeline`, `TextFilter`); port to .NET 3.5. (`TextFilter` reworked for NGUI bracket markup.)
- [x] Pre-load `Tolk.dll` (companions co-located) at `Awake` so the `DllImport` resolves; log clearly if it doesn't.
- [x] In `Awake`: initialize logging and speech, log `[HoFAccess] loaded`, and speak a startup line.
- [x] **Proof:** launch the game and get BOTH — a `[HoFAccess]` line in `output_log.txt` / BepInEx console AND a spoken startup line through the screen reader. Full output chain working end to end, with logging and speech both in place. (Verified: log chain present, "Hand of Fate Access loaded" spoken via JAWS, `card_table` scene loads clean with zero serialization errors.)

## Phase 2 - Focus spine (prove the core bet)
Split every focus readout into two layers so logic stays testable: a thin Unity **adapter** (in the plugin) that extracts raw state from the focused GameObject and does NO formatting, and **Core** composition that turns that raw state into the spoken string. The adapter is minimal and validated only in-game; the composition is unit-tested. Keep Unity types from leaking past the adapter boundary.
- [x] Stand up the validated manual Harmony-patch helper (logs success/failure per patch — see Cross-cutting) and use it for this and all later patches. (`Patching/HarmonyPatcher`.)
- [x] Harmony-postfix `UICamera.SetSelection`; on selection change in the Controller scheme, store the focused GameObject and set a dirty flag. Never speak from the hook. (`Patches/UICamera_SetSelection_Patch` -> `Focus/FocusTracker`; scheme-scoped to Controller, which NGUI also uses for keyboard arrow nav.)
- [x] Adapter: pull raw text from the focused object (`UILabel.text` on it or its children) into a plain DTO defined in Core (e.g. focused-control name plus its label texts). No Unity types in the DTO. (`Focus/FocusAdapter` -> `Core/Focus/FocusDto`.)
- [x] Core: compose the announcement from the DTO and unit-test the composition (empty labels, multiple child labels, markup via `TextFilter`). (`Core/Focus/FocusComposer`, `FocusComposerTests`.)
- [x] Announce once per frame from the update loop via the dirty flag; route through `SpeechPipeline`. (`Plugin.PumpFocus`, interrupt mode since navigation supersedes.)
- **Proof:** navigating the main menu with keyboard/controller speaks each focused item.
- **Architectural proof:** the focus-to-text composition is covered by offline tests; only raw extraction needs the game.

## Phase 3 - Proxy model + full readouts
- [x] `UIElement` base + `ProxyFactory` generic fallback (read all child labels, not just the first). (`Core/UI/UIElement.cs` with `GenericElement`; `Focus/ProxyFactory` is the single dispatch point, resolving the owning `Card` via `GetComponentInParent`.)
- [x] `Container` hierarchy + `FocusContext` path-diffing (announce only changed container context). **Realized one layer up at the screen level:** NGUI container/panel names are developer-internal, but mod-authored screen names are clean, so "announce only the changed context" now lives in `ScreenStack`'s entry-diffing (`Core/Screens/ScreenStack.cs`) keyed on the game's `GameState` machine rather than on nameless NGUI containers.
- [x] `Message` system (composable, markup-stripped at resolve). (`Core/UI/Message.cs`: ordered raw parts, filtered/empty-dropped/deduped/joined at `Resolve`.)
- [x] Targeted proxies for the high-value controls: encounter cards (name + full text + costs + tokens), equipment, dealer prompts. Use View classes for shared game-model reads. (One `CardElement`/`CardInfo` covers all `Card` subclasses - encounter, equipment, gold/food/health stat counters; encounter cards also read token stakes and completion. Dealer prompt is the encounter card's `Description`. Fame is not an in-run resource. No separate View needed since one card reader suffices; add View classes when a screen shares model reads across controls.)
- [x] `ScreenManager` (built as the `Screens` layer) — screen detection driven by the game's `GameState` machine, a Core `ScreenStack` with entry-announcement diffing, encounter/combat/shop overlays edge-detected from singletons, and the screen name spoken on entry. (`Core/Screens/{ScreenId,ScreenCatalog,ScreenStack}.cs`, `Screens/GameScreenWatcher.cs`.) Overlays that live outside the GameState machine are detected the same way: the pause menu (`PauseMenuManager` state) and modal dialogues (`MenuManager` Dialogue stack), the latter reading the live prompt text via `DialogueReader`/`DialogueInfo` so a modal is never silent. The stack is exposed for Phase 4 input dispatch and Phase 5 focus override to consume; the per-screen focus-resolution override and input-claim seams are defined but not wired (no dead dispatch). `ProxyFactory.Create` remains the focus resolver until Phase 5 needs the override.
- **Proof:** a focused card/equipment reads its complete information, correctly, across the main game screens; each screen names itself on entry and modal/pause overlays announce. *(Confirmed in-game; passed a multi-reviewer code pass whose findings - the queued-focus latch, nested dialogues, selection-forwarding focus, the gain/lose wording sitting in the adapter, and the catch re-touching a destroyed object - are fixed.)*

## Phase 4 - Input handler
The mod needs its own keys for things the game has no control for (map cursor, on-demand status lookups). This phase builds the input layer those depend on. (Review buffers and an F1 help overlay were considered and cut as unnecessary: the game's own controller navigation plus the focus readouts already cover review, and dedicated status keys serve the lookup need more directly.)
- [x] Input layer: an `InputRouter` polled each frame that dispatches to registered bindings, with `ButtonAction` (edge-triggered key / controller-button actions, any of several triggers firing one action) and a per-binding activation gate (`Func<bool>`) for screen scoping. Not rebindable (the actions are few, so hardcoded keys); each binding's `Poll` runs inside its own try/catch so one bad binding logs rather than stranding all input. The unclaimed-key fall-through is only needed where a mod key collides with the game's own (the map's Ctrl+arrow), so it is built with the map cursor in Phase 5 rather than speculatively here. (`Input/{IInputBinding,InputRouter,ButtonAction,Gamepad}.cs`.)
- [x] Status key: speaks the player's resources on demand without moving focus, bound to `/` and right-stick click (both verified unbound by the game's keyboard and controller tables, so no fall-through needed). Reuses the existing `ResourceReader` adapter and Core `ResourceReadout.ComposeStatus`; off a run it speaks a "No run in progress" line so the key is never silent. (`Resources/ResourceStatus.cs`.)
- [x] Map-explorer navigation keys (the Phase 5 consumer): `DirectionalAction`, a repeating four-way binding reading right stick on controller and Ctrl+arrows on keyboard, scoped to the map screen. The right stick is free on the map (its only right-stick uses are shop item-rotate, deck-builder archetype-panel scroll, and credits scroll, none overlapping the map). Plain arrows still drive the game's own UI nav; Ctrl+arrows are consumed in the `UISelectable.OnKey` patch (now a bool prefix) so the game's selection does not also move. (`Input/{Direction,DirectionalAction}.cs`, `Maps/MapInput.cs`.)
- **Proof:** a bound mod key fires its action *(status key confirmed in-game on `/` and right-stick click; map cursor moves on Ctrl+arrows and right stick)*; the unclaimed-key fall-through is proven by Ctrl+arrows moving only the cursor while plain arrows still drive the game's nav, both confirmed in-game.

## Phase 5 - Map explorer
A sighted player perceives the whole board at once: their position (the `PlayerCounter` token), the full layout and 4-directional orthogonal adjacency (no diagonals), and which cards are reachable now. The game's own selection cursor is local, though: it can only land on unlocked slots adjacent to the player and cannot even reach a distant card. So a **free-roam cursor decoupled from movement** is required to give a blind player that global read; it inspects only. Moving is left to the game's own selection (plain arrows step among reachable slots and already announce, then confirm activates), so the two cursors never fight. (Approach and the facts below confirmed this session via two temporary in-game probes; see the `map-board-cards-face-down-by-default` memory.)
- [x] Capture the map-cursor input scoped to the map screen via the `DirectionalAction` binding (Phase 4): right stick on controller, Ctrl+arrows on keyboard, the arrow consumed in the `UISelectable.OnKey` patch so the game's nav doesn't also move.
- [x] Free-roam cursor over `Map.Instance.MapLayout`, opened anchored on the player's slot (`PlayerCounter.CurrentSlot`) each time the map becomes active. **Walks card to card, not a grid sweep:** a step lands only on a neighbouring slot (`GetSlot` non-null), so it never sits on a gap. The playable board is 4-connected (the player walks it on foot), so card-walking still reaches every slot. While on the map, the free cursor snaps onto the game's own selected slot whenever the player moves the game cursor (plain arrows), so the two stay together; our Ctrl+arrow moves are swallowed before the game's selection changes, so free-roaming never triggers that snap. (`Maps/{MapBoardReader,MapCursor}.cs`.)
- [x] Per-card readout reuses `MapSlotReader`, already corrected for the board (face-up = `Flipped XOR (Complete || Revealed)`; otherwise withheld as face-down). Adds the state the card alone does not carry: here (player) / completed (carried by the card itself) / reachable (`IsUnlocked` and adjacent to the player) / open (unlocked, not adjacent) / locked. **In place of an absolute position, it reads the exits** (the directions that hold a neighbouring card), so the player learns the board's shape by walking it. Composition is Core `MapCellReadout` over the `MapCellInfo`/`MapExits` DTOs, unit-tested. (`Core/UI/MapCellInfo.cs`, `Core/Screens/MapCellReadout.cs`.)
- **Proof:** a blind player can survey the whole board (layout, adjacency, each card's identity, state, and exits) and move through it without sight. *(Confirmed in-game: the cursor walks the board card to card, reads each card, follows the game cursor, and moves on both the stick and Ctrl+arrows; moving the player uses the game's own selection, which already announces.)*

## Chance gambit minigame (unplanned, shipped)
The chance-card shuffle is a visual shell game with nothing for focus-reading to announce, so it got a bespoke audio treatment.
- [x] Identity tones: each card slot gets a distinct synthesized instrument voice (organ, violin, plucked guitar, bell), panned across the stereo field by slot. (`Core/Gambit/GambitTones`.)
- [x] Spoken outcomes: the game's own localized `CHANCE_TITLE_*` titles, rendered per slot through a vendored native x86 SAPI shim (`native/hofsapi/`) because Tolk returns no audio buffer to pan or sequence. SAPI is used ONLY for the Establish teaching walk; everything else in the minigame speaks through the normal screen-reader pipeline.
- [x] Choreography: an Establish walk (freezes time, teaches each slot's tone and outcome), live shuffle following (each card's sustained tone pans by its live slot index), a silent Select phase that speaks "Slot N" on focus, the pick reveal, and the guardian-angel keep-or-redraw. (`Gambit/GambitWatcher`.)
- **Proof:** the shuffle is trackable by ear (a randomized prototype test was called 5 of 5) and the full minigame is confirmed in-game.

## Phase 6 - Combat (in progress)
Built on audio, not focus-reading. The subsystem was built fresh on a Unity AudioSource voice pool rather than lifting oni-access's audio engine (Unity 5.3's audio IS FMOD already; an external lib would add a second device path with no game-mixer integration): Core seam `Audio/{IAudioBackend,AudioEngine,VoicePool}` + plugin `UnityAudioBackend`, procedural synthesis in Core (unit-tested), samples normalized to mono 44100 16-bit WAV at -3 dB peak. Hook signatures and verified game mechanics (reflect arming, dodge i-frames, attack timing) live in the memory files, not here. Big picture: the reactive layer (what is attacking me and what do I do) is functionally complete pending live tuning; the environmental layer (what is on the ground and where do I go) is built for movers and ground areas pending live validation, with beams, segment chains, and the objects layer the remaining construction.
- [x] Wall awareness: four directional wind loops that swell as geometry closes in (measured against the NavMesh edge, which is what actually bounds combat movement, not physics colliders), plus a panned bump with footstep suppression when the player walks into geometry. Validated live.
- [x] Projectile tracking: every enemy projectile in flight gets its own synthesized voice; pan = east/west, pitch = north/south, volume = distance; a reflected shot tumbles audibly faster. Validated live.
- [x] Attack telegraphs: one classified cue per attack at parry-window open, the same instant as the sighted player's flash (clang = press block now and it connects, crack = move). Standard enemies, boss basics, and turrets validated live; the bespoke boss attacks (Ogre, Reaper, Lich, Shaman, Kraken, Ratman King, Hermit bomb) are built on verified hooks but not yet heard in a fight.
- [x] Audit instruments (log-only, run even if audio fails): every player hit logs its source attack and seconds since the last cue; every non-projectile hazard spawn logs type, team, and position. The recon feed for hazard voicing.
- [x] Mover hazards (lobs, lightning heads): from the player's side these ARE projectiles (incoming thing, dodge away), so they ride the existing projectile voice rather than new sound design; they never enter `CombatManager.Projectiles`, so each class's own `OnEngage` override feeds the sonifier, with the voice stopping at the proxy's expire flag, not its destroy. Each launch also fires the dodge cue at the thrower, deduped per attacker (Core `MoverCueGate`, one cue per source per window) so an attack that already telegraphed (Hermit throw, boss Begin, ranged chokepoint) never cracks twice. Built on verified hooks; not yet heard in a real fight.
- [x] Zone hazards, ground areas (the prototype): one synthesized danger-zone voice, a pulsed buzzy tone (the mix's only tonal pulse, so it cannot be confused with the wind or the tumble), placed at the zone's nearest dangerous point so "move away from the sound" is always the correct escape for every shape (disc edge, ring band, even the safe hole). State, not class, carries urgency: arming throb (leaving is free), active buzz, inside rattle at full volume; pulse rates spaced ~3x so the bearing pitch-shift cannot blur states. Tracked by polling the game's own `CombatProxyArea.AllAreas` (no hooks); nearest 3 voiced; cones over-warned as discs; a growing zone warns its full authored footprint from frame one; mines voiced active from birth (the arming delay only defers a mine's fuse, and a tripped mine allows no safe exit, so it is never presented as dormant). The engage recon log now also dumps each area's authored tuning (delay, grow, timeout, ring, angle, mine), so every fight banks the data for refinement. Prototype-first by design: testing happens in fights the voice already protects, not recon fights without it. Sound design auditioned and approved offline from mockups rendered by the shipped synthesis; survived a decompiled-source assumptions check and a code review.
- [x] Trap voicing: the level's traps (swinging blades, spike floors - a separate authored system, not CombatProxy) join the zone voice as a second source, same loops and nearest-point grammar, so the learned vocabulary transfers. Traps have no registry: scanned once per encounter (`FindObjectsOfType`, keyed on `CombatEncounter.Instance` changing), then the damage trigger's apply flag and collider are read live each frame. The voice simply following the armed flag makes the trap's deterministic phase cycle audible as a rhythm, so the player times the crossing by ear; the safe beat voices as the arming throb (damage off, crossing free) and inside still rattles on safe beats because arming is retroactive (going hot hits everyone already standing in it). A trap whose cycle idles on a proximity trigger gets the one new state, primed (1 Hz hard pulse: damage off but approaching fires it), so it never reads as absent. Built on decompiled-source mechanics; not yet heard in game.
- [ ] Zone hazards, remaining shapes: beams (hook `CombatProxyBeam.Engage`, which hands over the full line geometry; nearest-point-on-segment math joins Core) and trail/lightning segment chains (voiced at the nearest segment). Then tune all the loops against live play and the recon distributions.
- [ ] Live validation pass for the new hazard audio, in the next real fights: movers - lob and lightning flight voices track by ear and the launch crack fires exactly once per attack (the Hermit bomb is the known double-cue case the gate must suppress); zones - the three states read correctly in play, the inside rattle lands at the true edge, bearings match, and the nearest-3 cap holds in cluttered arenas; traps - the armed/safe alternation is followable enough to time a crossing, and a primed trap is audible before it fires. Every fight also banks recon for the tuning round: the area tuning dumps and the hit tripwire's uncued-damage lines.
- [ ] Boss cue timing tuning: read the hit-tripwire deltas after boss fights and move any cue that fires uselessly early onto a tighter in-animation event. Open check the logs will answer: whether Kraken-fight cues survive the `AttackCues` pump's `CombatEncounter` gate.
- [ ] Objects layer: chest locator, exit orb (trap armed/safe states shipped with the zone voice above). Nothing built; the exit orb matters most - it is how the player leaves every cleared room. `TrapChest` / `TrapExit` (player-only walk-in volumes; `TrapExit` has a public `OnComplete`) are the natural seams in trap rooms.
- [ ] The Dealer fight, last, as its own milestone. Fully bespoke - none of the generic machinery covers him, though the hooks are already verified: hand-swipe cue, missile quick-time event reading (which button; 0.5 s counter / 1.0 s dodge windows), rotating beam tracking, pulse warning, vulnerability-phase announcements.
- [ ] Optional, lowest priority: on-demand enemy locator key.
- **Proof:** a representative fight is winnable without sight.
